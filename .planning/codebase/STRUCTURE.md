## Directory Purposes

**core/:**
- Purpose: Contains the heart of the application's logic, including bot event handling, configuration management, database interactions, and internationalization.
- Contains: Python modules (`.py` files) and data files (`.json`).
- Key files: `core/main.py`, `core/handlers.py`, `core/config.py`, `core/database.py`, `core/i18n.py`, `core/responses.json`.

**data/:**
- Purpose: Intended for storing persistent application data. This might include a SQLite database file, temporary files, or other data that the application generates or uses.
- Contains: Data files (e.g., `.db`, images, temporary files).
- Key files: Specific data files would reside here as they are created or used.

**docs/:**
- Purpose: Houses all project documentation, including higher-level descriptions, architectural diagrams, or specific feature explanations.
- Contains: Markdown files (`.md`), potentially other document formats.
- Key files: `docs/superpowers/plans/` (contains specific plans).

**.planning/:**
- Purpose: Stores artifacts generated by the GSD (Gemini Software Development) agent, such as codebase analysis documents.
- Contains: Markdown files (`.md`).
- Key files: `ARCHITECTURE.md`, `STRUCTURE.md`, `STACK.md`, `INTEGRATIONS.md`, etc.

**venv/:**
- Purpose: Standard Python virtual environment. It isolates project dependencies from other Python projects.
- Contains: Python executables, libraries, and site-packages for the project.
- Key files: `venv/bin/python`, `venv/lib/pythonX.Y/site-packages/`.

## Key File Locations

**Entry Points:**
- `main.py`: The top-level script that kicks off the application by calling `core.main.main()`.
- `core/main.py`: The actual main function that initializes the Telegram bot, registers all handlers, and starts the polling loop.

**Configuration:**
- `core/config.py`: Python module for loading and managing environment-based configuration variables.
- `requirements.txt`: Lists all Python package dependencies for the project.
- `docker-compose.yml`: Defines the services, networks, and volumes for a Docker-based multi-container application.
- `Dockerfile`: Instructions for building a Docker image of the application.

**Core Logic:**
- `core/handlers.py`: Contains the implementations for all the bot's commands and message handlers.
- `core/database.py`: Manages the application's database connection and initialization.
- `core/i18n.py`: Provides functions for internationalization (i18n) support.
- `core/parsing.py`: Contains utility functions, possibly for parsing specific data formats or performing OCR.

**Documentation:**
- `README.md`: General project overview and setup instructions.
- `ROADMAP.md`: Project development roadmap.
- `GEMINI.md`: Specific documentation related to Gemini AI integration or features.
- `docs/superpowers/plans/`: Directory containing detailed plans or specifications.

## Naming Conventions

**Files:**
- **Python Modules:** `snake_case` (e.g., `main.py`, `config.py`, `handlers.py`).
- **Markdown Files:** `PASCAL_CASE.md` or `snake_case.md` (e.g., `README.md`, `GEMINI.md`, `ARCHITECTURE.md`).

**Directories:**
- `snake_case` (e.g., `core/`, `data/`, `docs/`).

**Variables/Functions (in Python):**
- `snake_case` is expected for function and variable names, following Python's PEP 8 guidelines.

**Classes (in Python):**
- `PascalCase` is expected for class names, following Python's PEP 8 guidelines.

## Where to Add New Code

**New Feature (e.g., a new bot command):**
- Primary logic: Add a new function to `core/handlers.py` for the command's implementation.
- Registration: Register the new handler in `core/main.py` using `app.add_handler()`.
- Localization (if applicable): Add new entries to `core/responses.json` and retrieve them via `core/i18n.py`.

**New Component/Module (e.g., a new external API integration):**
- Implementation: Create a new Python file in the `core/` directory (e.g., `core/external_api.py`).
- Integration: Import and use this new module within `core/handlers.py` or `core/main.py` as needed.

**Utilities:**
- Shared helpers: Add new functions to `core/parsing.py` or create a new dedicated utility module in `core/` if the functionality is distinct and substantial.

## Special Directories

**venv/:**
- Purpose: Contains the isolated Python environment and its dependencies.
- Generated: Yes (typically by `python -m venv venv`).
- Committed: No (listed in `.gitignore`).

**.git/:**
- Purpose: Stores all the information about the project's version control.
- Generated: Yes (by `git init`).
- Committed: Not directly, it's the repository itself.

**.planning/:**
- Purpose: Holds documentation generated by automation tools like GSD.
- Generated: Yes (by the GSD agent).
- Committed: Yes, as these are valuable project artifacts.

**data/:**
- Purpose: For persistent data storage.
- Generated: Potentially (e.g., database files created by the application).
- Committed: Depends on the specific data; often `.gitignore` for large or frequently changing data (e.g., SQLite `.db` file). Small, static data might be committed.
